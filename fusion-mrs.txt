Forecourt Controller Pump 1 Pump 2 Fore court GUI Console Web Administration Message Router Forecourt Controller Pump 1 Pump 2 Fore court GUI Console Web Administration Conexión Todos los clientes deben establecer una o más conexiones con la IP de Fusion y el puerto 3011. Esa conexión debe permanecer abierta si se van a recibir mensajes no solicitados. La conexión debe mantenerse viva con al menos un mensaje cada 150 segundos. Si no hay actividad, el cliente debe enviar un mensaje ECHO (descrito a continuación). Si Fusion no detecta actividad después de 360 segundos, la conexión se cierra y el cliente tendrá que volver a conectarse. Formato del mensaje Todos los mensajes hacia / desde el enrutador de mensajes tienen un formato común. El mensaje se compone de todos los bytes ASCII. El formato es el siguiente: <len>|<crypt format>|<message version>|<user id>|<message type>|<event type>|<destination>|<origin>|<parameters... >|^ Descripción de los campos <len> Campo de longitud fija de 5 bytes que contiene la longitud del mensaje desde la versión <menssage> hasta el ^ incluido. <formato criptográfico> Campo de longitud fija de 1 byte que describe el formato de cripta del mensaje. El valor de '5' significa que no hay cifrado. El valor de '6' significa cifrado. Consulte el capítulo Cifrado. <versión del mensaje> Campo de longitud fija de 1 byte. Valor fijo '2'. < identificador de usuario > Campo presente solo en la versión de mensaje 2 o superior. Este es el ID de usuario registrado en la conexión que solicita el comando. Déjalo en blanco. Completado automáticamente por Fusion. <tipo de mensaje> Hay 6 tipos diferentes de mensajes. Estos son: SUSCRIBIRSE : Solicitud de suscripción a uno o más eventos no solicitados  DARSE DE BAJA : Solicitud de cancelación de suscripción a un evento previamente suscrito  POST : Publicar una solicitud, respuesta o evento. MUERTO : Dígale al router que almacene este mensaje y lo envíe cuando la conexión real se caiga. El mensaje se enviará como un mensaje POST . Esto se utiliza principalmente para decirle a otros módulos que un módulo se cayó. EXIT : Fin de la conexión con el router ECHO : Mantiene viva la conexión <tipo de evento> Este campo es el que se describe en cada mensaje como "Nombre del evento". <destino> Si un módulo desea enviar una respuesta a otro módulo, incluirá aquí la identificación de destino del módulo. La identificación se compone de <dirección IP de destino>:puerto de <destino>. Este campo puede estar en blanco si el mensaje no tiene un destino en particular. <origin> Todos los mensajes incluirán el origen en el mismo formato del <destino>. Al enviar un mensaje, debe dejar este campo en blanco y el enrutador de mensajes lo configurará por usted. <parámetros... > Un mensaje puede contener 0, 1 o muchos parámetros. El formato de los parámetros es <nombre del parámetro>=<valor del parámetro>. Si se requiere más de un parámetro, entonces están separados por una tubería "|". Encriptación Si el mensaje está formateado con cifrado, entonces debe usar el siguiente algoritmo para cifrar y descifrar los mensajes. Todos los mensajes contendrán formato <len>|<crypt>| sin cifrado. Después de eso, todo el resto está encriptado. El cifrado consiste en un XOR con una máscara para cada byte. La máscara comienza con 0xAB y agregará 9 después de cada byte. Entonces, para el primer byte cifrado, la máscara será 0xAB, para el segundo, 0xB4, para el tercer 0xBD y así sucesivamente. Para su referencia, el código C se proporciona como ejemplo: void easyCrypt ( char * p_pMessage , int p_iLen ) { unsigned char mask = 0xAB; int i ; for ( i =0; i < p_iLen ; i ++, mask += 9) { p_pMessage [ i ] ^= mask ; } } Datos confidenciales Fusion Message Router puede estar registrando los mensajes intercambiados en un archivo local para fines de depuración y solución de problemas. Algunos de estos mensajes pueden contener datos confidenciales que nunca deben registrarse (datos de tarjetas de crédito / débito, datos de banda magnética, etc.). Para lograr esto, el mensaje puede contener un parámetro adicional llamado "SENS" con instrucciones sobre cómo registrar los parámetros contenidos en este documento. Si el valor de ese parámetro "SENS" es "SÍ", entonces todos los parámetros del mensaje actual se consideran confidenciales, y ninguno de ellos se registrará. Por otro lado, puede establecer como el valor del parámetro "SENS", el nombre del único parámetro sensible de ese mensaje. En este caso, ese parámetro no se registrará, pero sí todos los demás. Si falta el parámetro "SENS", o con cualquier otro valor, se ignorará. Eco Para mantener viva la conexión, debe haber mensajes que vayan y vengan cada 150 segundos. El tipo de mensaje ECHO no necesita un "Tipo de evento" ni parámetros. Este eco será enviado de vuelta al cliente por el router con el mismo formato. Eventos Los eventos son generados por diferentes módulos sin necesidad de hacerlo, y cualquier otro módulo puede suscribirse para recibirlos. Suscribirse a eventos es la forma correcta de obtener información actualizada de Fusion. Se desaconseja el enfoque de sondeo, ya que degrada el rendimiento del sistema y la información disponible no podría considerarse definitiva. La información adicional como la identificación del asistente no se procesa al mismo tiempo que la venta de la bomba, por ejemplo. Es posible que las solicitudes entre estos dos eventos no reciban la identificación del asistente. Format is YYYYMMDD TI Time of closure. Format is HHMMSS Suscripciones Para comenzar a recibir los eventos descritos, un cliente primero debe suscribirse a ellos. La suscripción solo funcionará para EVENTOS , pero no para SOLICITUDES , RESPUESTAS y mensajes de SEGURIDAD . Un cliente puede suscribirse a ninguno, uno o varios eventos. El mensaje de suscripción puede contener un comodín al final de la descripción del EVENTO, para suscribirse a un grupo de EVENTOS. Un Cliente puede suscribirse a todos los eventos suscribiéndose al evento "TODOS". EJEMPLOS: La suscripción al evento "EVT_PUMP_STATUS_CHANGE_ID_003" recibirá eventos de cambio de estado de la bomba SOLO desde la bomba número 3. La suscripción al evento "EVT_PUMP_STATUS_CHANGE_ID_*" recibirá eventos de cambio de estado de bomba de TODAS las bombas. La suscripción al evento "EVT_PUMP_*" recibirá todos los eventos generados a partir de las bombas.  La suscripción al evento "ALL" o "*" recibirá TODOS los eventos de todos los módulos. Un cliente también puede darse de baja de uno o más eventos de la misma manera que lo hace con la suscripción. Parámetros opcionales de suscripción: Para rellenar la tabla de base de datos de acciones de seguridad, la suscripción puede incluir dos parámetros opcionales. El primer parámetro es el "SEC" con un valor fijo de "1". Esto le dirá al módulo de seguridad que este tipo de EVENTO se incluirá en el esquema de seguridad, por lo que cada vez que aparezca un módulo este tipo de evento, será verificado por el complemento de seguridad. Solicitudes Las solicitudes son mensajes direccionales para solicitar información o acciones de un módulo específico. Algunas solicitudes recibirán una RESPUESTA con la información o el estado esperados. Si se envía una SOLICITUD y no hay ningún módulo suscrito a ella para cumplir con el requisito, se enviará un RES_EVENT_NOT_SUBSCRIBED para que el cliente sepa que ningún módulo ejecutará esa acción. Las RESPUESTAS a una SOLICITUD solo son recibidas por el originador de la SOLICITUD , sin importar si el originador no está suscrito a esa RESPUESTA. Dado que algunas acciones están restringidas para algunos usuarios (seguridad), si el módulo que envía la SOLICITUD no ha iniciado sesión con un usuario habilitado para ejecutar esa SOLICITUD , recibirá un mensaje de seguridad como RES_SECU_ACCESS_DENIED .
